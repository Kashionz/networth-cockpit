## 1. 產品快速理解

### 1.1 一句話產品定位

> 把分散在各處的資產彙整在一個地方,看見配置現況與淨值成長軌跡,並透過規則引擎與 AI 提供貼著個人風險屬性的調整方向(教育性質,非投資建議)。

### 1.2 用戶畫像

**主要用戶**:28–42 歲的台灣上班族投資人,月薪 6–15 萬,信用卡全額繳清、有緊急金、月儲蓄率 15–35%。

**心態關鍵字**:自律、時間有限、不焦慮、不炫富、想看到結構而非細節、想要控制感而非愧疚感。

**他們不是誰**:不是高槓桿玩家、不是當沖客、不是高負債需要被救的人、不是焦慮過度想記每杯飲料的人。

### 1.3 為什麼這影響設計

設計這個產品的時候請想像一個情境:**一位週日晚上喝著咖啡、想看看這個月過得如何的工程師**。不是一位追著紅綠 K 線、需要即時警報的當沖客。所有視覺決策都應該服務這個情境。

## 2. 設計哲學的硬約束

> **本章是整份文件的核心。每一條都源自 PRD 第 4 章「產品原則與記帳哲學」。違反這些約束的設計會直接傷害產品。**

### 2.1 五條硬約束

**約束一:儲蓄率優先於總資產**

主畫面第一個視覺焦點應該是「本月儲蓄率」,不是「總資產」。儲蓄率對長期淨值成長的影響遠大於配置優化,這個產品的價值排序必須在 UI 上反映出來。

**約束二:中性,不羞辱**

- ❌ 不准用紅字標示「超支」、「警告」、「危險」
- ❌ 不准用驚嘆號、警告 icon(⚠️)、紅色 badge
- ❌ 不准做「連續記帳天數」、「連勝紀錄」、Duolingo 式的成就壓力
- ❌ 不准在月底結算時用任何「失敗」、「未達標」之類的負面 framing
- ✅ 用「建議」、「可考慮」、「值得檢視」這類中性詞
- ✅ 用色彩深淺與位置層級表達輕重緩急,而不是情緒色

**約束三:向前看,不向後評判**

預算制(向前看 + 規劃)優於記帳制(向後看 + 評斷)。視覺上:

- 預算進度條要做得有「掌控感」而非「監視感」
- 月度報告的 framing 是「下月可以怎麼分配」而非「上月你哪裡花太多」
- 不做「比上月超支了 X 元」這種比較性紅字數字

**約束四:摩擦優先**

每一個要求使用者輸入的欄位、每一次點擊、每一個彈窗,都要回答「不問可以嗎?」設計上:

- 預設值要聰明(基於上月、基於模板)
- 表單要短(分步驟而非一頁問完)
- 信用卡帳單匯入流程必須**極度順滑**(這是產品命脈)
- 不做「歡迎引導動畫超過 3 屏」

**約束五:抓大放小**

不展示瑣碎細節,展示結構與軌跡:

- 不做「7-11 飲料 45 元」這種逐筆消費長列表為主畫面
- 月度報告以類別、預算達成率、淨值變化為主,不以交易明細為主
- 圖表以「能在手機上 1 秒看懂」為標準

### 2.2 「不做」清單(刻意的)

以下功能與視覺元素**明確不做**,即使技術上做得到、即使其他理財 App 都這樣做:

| 不做的事 | 理由 |
|---|---|
| 紅色超支警告 | 違反約束二 |
| 連續記帳天數 / streak | 違反約束二 |
| 月度排行榜 / 與其他用戶比較 | 違反產品定位(不炫富不攀比) |
| 詳細多級消費分類學(餐飲→早餐→西式→咖啡) | 違反約束五 |
| 拍發票辨識單筆消費 | 違反約束四 |
| 每日記帳推播提醒 | 違反約束四 |
| 「省錢挑戰」、「7 天無花費挑戰」 | 違反約束二 |
| 紅綠 K 線、即時報價跑馬燈 | 違反產品定位(這不是看盤工具) |
| 大量 emoji 點綴 | 違反產品調性(成熟、低調) |
| 註冊時要求填過多個資 | 違反約束四 |

### 2.3 整體調性關鍵字

- **成熟、低調、可信任**(避免幼稚化、避免過度遊戲化)
- **平靜、克制**(避免焦慮、避免緊張感)
- **有掌控感**(讓使用者感到「我在管理我的錢」而非「我在被錢管理」)
- **隱密、安全**(處理的是敏感金錢資訊,視覺上要傳達這種尊重)

## 4. 核心頁面資訊架構

### 4.1 `/` Dashboard(優先序 #1)

**目的**:讓使用者在 10 秒內看到「這個月過得如何」。

**資訊區塊**(由上至下,優先序由高至低):

```
┌─────────────────────────────────────────────┐
│ Block 1:本月儲蓄率                           │
│   - 主數字:本月儲蓄率 28%                    │
│   - 副資訊:目標 30%、達成度視覺化            │
│   - **產品的第一視覺焦點**                   │
├─────────────────────────────────────────────┤
│ Block 2:淨資產                               │
│   - 主數字:淨資產 NT$ 2,450,000              │
│   - 副資訊:較上月 +85,000 (+3.5%)            │
│   - 走勢縮圖(過去 6 個月)                    │
├─────────────────────────────────────────────┤
│ Block 3:本月預算進度                         │
│   - 三條進度條:固定/生活/彈性                │
│   - 用量比例 + 剩餘天數                       │
├─────────────────────────────────────────────┤
│ Block 4:投資配置                             │
│   - 三類別比例:股 / 債 / 現                   │
│   - 偏離目標的指示                            │
├─────────────────────────────────────────────┤
│ Block 5:健康警示(若有)                      │
│   - 0–N 張卡片,使用色彩深淺表示優先級         │
│   - **絕不使用紅字驚嘆號**                    │
├─────────────────────────────────────────────┤
│ Block 6:本期信用卡帳單(若臨近結帳日)         │
│   - 預估金額、結帳日、繳款日                  │
└─────────────────────────────────────────────┘
```

**設計需求**:
- 每個 Block 在手機上應為「一指可滾過」的高度
- Block 1 必須在 fold 上方完整可見
- 所有金額顯示應使用等寬數字,讓位數對齊
- 如使用者開啟「隱私模式」(見 9.3),所有金額替換為 ¥¥¥,但結構保持

### 4.2 `/transactions/import` 信用卡帳單匯入(優先序 #2)

**目的**:讓使用者每月只花 5 分鐘完成全月記帳。**這個流程的順滑度直接決定產品成敗。**

**完整流程**:

```
Step 1:選擇要匯入的卡片
  - 列出已建立的信用卡
  - 顯示「上次匯入到 OO 月」

Step 2:上傳檔案
  - 支援拖拉、點擊上傳
  - 接受:CSV、PDF、圖片(Phase 1)
  - 上傳中顯示進度

Step 3:解析中
  - 顯示「正在解析 N 筆交易」
  - 視覺上必須讓使用者感到「事情正在進行」

Step 4:審核分類結果(關鍵頁面)
  - 列出所有解析結果
  - 每筆顯示:商家、金額、分類、信心度
  - 已自動分類的(信心度高)標示為「已分類 N 筆」、預設摺疊
  - 待審核的(信心度低或新商家)展開、需要使用者確認
  - 批次操作:一鍵接受全部、一鍵改分類

Step 5:確認寫入
  - 顯示總覽:「將寫入 N 筆,影響預算 X / Y / Z」
  - 完成 → 跳轉回 Dashboard 並顯示「本月帳單已匯入」
```

**設計需求**:
- Step 4 是設計重點。目標是讓使用者**不需要逐筆點擊**,而是用「批次掃過」的方式完成審核
- 「已自動分類」與「待審核」必須有清楚的視覺區隔(深淺、留白、區塊化)
- 使用者改分類的操作要極度輕量(下拉、tag 點擊),不要彈窗
- 新商家的分類確認後,系統會記住規則,設計上要傳達這個「越用越快」的感覺

### 4.3 `/budget` 本月預算(優先序 #3)

**目的**:讓使用者在月中看到「我還能花多少」、月底看到「下個月可以怎麼分配」。

**資訊區塊**:

```
Block 1:本月總覽
  - 月薪、總可支配、已分配、儲蓄目標

Block 2:三類預算進度
  - 固定支出:已扣除多少 / 剩餘
  - 生活預算:用量條(80% 進警示區,但不用紅色)
  - 彈性預算:用量條
  - 每條都附「日均」、「剩餘天數可用日均」

Block 3:本月已記錄的大額支出
  - 列表,僅顯示 5,000 以上的項目
  - 點擊可看詳情

Block 4:下月預算規劃(月底時顯示)
  - 「下月想調整分配嗎?」
  - 基於本月達成率的建議
```

**設計需求**:
- 預算用量條的視覺感是這個頁面的關鍵。必須傳達「掌控感」而非「警示感」
- 即使用量達 95%,也不該讓使用者感到焦慮 — 用色階深淺而非紅字
- 月底時的「下月規劃」UX 非常重要,是把「事後評判」轉換為「事前規劃」的關鍵時刻

### 4.4 `/portfolio/allocation` 投資配置分析(優先序 #4)

**目的**:讓使用者看到現況 vs 目標的偏離,並理解該怎麼調整。

**資訊區塊**:

```
Block 1:配置概覽
  - 兩個並排的視覺化(現況 / 目標)
  - 各類別比例

Block 2:偏離度
  - 各類別偏離 ±%
  - 觸發 rebalance 提示時的視覺化

Block 3:持股明細(可摺疊)
  - 各類別下的個別資產
  - 集中度警示(單一個股 > 20%)

Block 4:補倉建議
  - 「下月可投入 N 萬時的比例分配方向」
  - 不指定具體標的(法規線)
```

**設計需求**:
- 「現況 vs 目標」的對比視覺化是這個頁面的核心。可考慮並排餅圖、堆疊長條、雷達圖等多種方案
- 偏離度的呈現必須清楚但不情緒化
- 「補倉建議」的免責聲明要清楚但不喧賓奪主

### 4.5 `/onboarding` 首次設定流程(優先序 #5)

**目的**:讓新使用者在 5 分鐘內完成設定,且不流失。

**步驟**:

```
1. Welcome
   - 一句話說明產品價值
   - 無動畫、無多屏教學

2. 風險屬性問卷(8–10 題)
   - 一題一頁 or 一題一卡
   - 進度顯示
   - 可跳過,後補

3. 目標配置設定
   - 顯示問卷結果建議
   - 允許微調
   - 可選「使用建議」一鍵帶過

4. 月薪輸入(用於預算建議)
   - 一個欄位
   - 解釋為什麼要問

5. 預算設定
   - 基於月薪套用 50/30/20 模板
   - 允許微調

6. 新增第一筆資產
   - 引導性,但可跳過
   - 完成後跳到 Dashboard
```

**設計需求**:
- 整體流程感受應該像「填一份簡單表單」,不像「過五關斬六將」
- 每一步都要有「跳過,稍後再說」的選項(摩擦優先)
- 進度視覺化要明確,讓使用者知道「還剩幾步」

### 4.6 其餘頁面(#6–#10)資訊架構摘要

**`/insights/[year-month]` 月度報告**
- 標題:當月淨值變化金額 + %
- 區塊:儲蓄率、預算達成、配置變化、健康警示總覽、AI 解讀文字、附本月里程碑(若有)
- 末尾:免責聲明

**`/assets` 資產總覽**
- 列表式呈現,以類別分組
- 每筆顯示:名稱、目前估值、佔比
- CRUD 操作

**`/cards/[id]` 信用卡詳情**
- 本期帳單金額、結帳日、繳款日
- 本期已記錄交易列表
- 歷史帳單列表

**`/settings/*` 設定**
- 標準的設定頁排版
- 隱私模式開關放醒目位置
- 資料匯出與帳號刪除清楚可見

**`/auth/{login,signup}`**
- 標準登入註冊
- 支援 Email + Google OAuth

## 6. 關鍵使用者流程

### 6.1 流程 A:首次進入(Onboarding)

```
未註冊使用者進入首頁
  → 點擊「開始」
  → 註冊(Email or Google)
  → /onboarding/welcome
  → /onboarding/risk-questionnaire(8 題)
  → /onboarding/target-allocation(可微調建議)
  → /onboarding/budget-setup(輸入月薪 → 套用模板)
  → /onboarding/first-asset(可跳過)
  → / Dashboard(空狀態,提示新增資料)
```

**設計重點**:
- 整體 5 分鐘內完成
- 每步皆可跳過
- 完成後 Dashboard 不應該是「空白挫折」,而是「期待充實」的 empty state

### 6.2 流程 B:每月信用卡帳單匯入(產品核心 flow)

```
信用卡結帳日 + 1 天
  → 推播:「OO 卡帳單已出,點此匯入」
  → 點擊推播 → /transactions/import?card=xxx
  → 上傳 CSV → 解析中(2–5 秒)
  → 審核頁:已自動分類 N 筆(摺疊)、待審核 M 筆(展開)
  → 使用者批次審核(目標:< 3 分鐘)
  → 確認寫入
  → 跳回 / Dashboard,Block 6 顯示「本月帳單已完成」
```

**設計重點**:
- 這個流程每月會走 1–4 次(看使用者有幾張卡)
- 每次都要順到使用者「不覺得在記帳」
- 若使用者拖延沒匯入,儀表板某處要有「N 張卡的 X 月帳單尚未匯入」的低調提示(不轟炸)

### 6.3 流程 C:月底回顧 + 下月規劃

```
每月最後一天 21:00
  → 推播:「OO 月已結束,看看回顧」(可關閉)
  → 點擊 → /insights/[this-month]
  → 看月度報告(淨值、儲蓄率、預算達成、配置變化、AI 解讀)
  → 末尾 CTA:「規劃下個月預算」
  → 點擊 → /budget?month=next
  → 顯示下月預算建議(基於本月)
  → 微調 → 確認 → 完成
```

**設計重點**:
- 月度報告的調性極度重要 — 這是使用者「對自己這個月下評語」的時刻
- 不論本月好壞,framing 都應該是中性、向前看的
- 即使儲蓄率不達標,也不應該有「失敗」感

### 6.4 流程 D:配置偏離 → Rebalance 提示

```
某日股市大漲後
  → 排程任務發現 user 的股票部位偏離目標 +12%
  → 推播一次(中性):「配置已偏離目標,可考慮回看」
  → 點擊 → /portfolio/allocation
  → 看到偏離視覺化 + 補倉建議方向
  → 不指定標的,只給比例
  → 使用者自行決定是否操作(系統不參與交易)
```

**設計重點**:
- 推播頻率必須節制(同一偏離事件 7 天內不重複推)
- 補倉建議必須有清楚的免責語

### 6.5 流程 E:大額手動記錄

```
使用者剛在 ATM 領了 5 萬給家人 / 用現金買了一台 5 萬的相機
  → 打開 App → /transactions/manual
  → 簡單表單:金額、日期、類別、來源帳戶、備註
  → 提交 → 跳回 Dashboard
```

**設計重點**:
- 這個流程必須極快 — 5,000 以上的支出每月可能 1–3 次
- 表單欄位順序與預設值要極度聰明
- 來源帳戶下拉應該預設「上次使用的」

### 6.6 流程 F:隱私模式切換

```
使用者在咖啡廳打開 App,旁邊有人
  → 主畫面右上角點擊「眼睛」icon
  → 所有金額立刻變為 ¥¥¥
  → 結構、比例、走勢仍可見
  → 再點一次回復
```

**設計重點**:
- 切換必須一鍵到位、無確認彈窗
- 隱藏狀態要有明顯的視覺指示,避免使用者忘記自己在隱藏模式

## 7. 設計系統需產出的資產

### 7.1 設計 tokens

- **色彩系統**
  - 主色 primary
  - 中性色階(背景層級、文字層級、邊框)
  - 狀態色:不使用傳統紅綠,而是建議用「藍 / 琥珀 / 紫 / 灰」的四階溫和色階表達 ok / 接近 / 警示 / 嚴重
  - 資產類別識別色(股/債/現金/加密)
  - 預算類別識別色(固定/生活/彈性)
- **字型系統**
  - 標題字型
  - 內文字型
  - **等寬數字字型**(關鍵,所有金額用)
- **排版尺度**:typography scale、line-height、letter-spacing
- **間距系統**:spacing scale(建議 4px / 8px base)
- **圓角、陰影、邊框**

### 7.2 元件庫

涵蓋第 5 章列出的所有共用元件,每個元件需包含:

- 預設狀態
- Hover / Active / Disabled 狀態
- 載入中狀態
- 空資料狀態
- 錯誤狀態(若適用)

### 7.3 Icon set

需要的 icon 類別:

- 資產類別:現金、銀行、股票、ETF、債券、加密、車輛、黃金
- 動作類:新增、編輯、刪除、匯入、匯出、隱私(眼睛)
- 導覽類:Dashboard、Assets、Budget、Insights、Settings
- 狀態類:健康警示等級(用形狀區分,不只用顏色)
- 分類類:預算分類、消費分類

**Icon 風格約束**:成熟、克制,不卡通化、不過度裝飾。建議 line / minimal 風格而非 filled / playful。

### 7.4 響應式斷點

- Mobile:< 640px(主要場景)
- Tablet:640–1024px
- Desktop:> 1024px

**Mobile-first 但 desktop 體驗同等重要**。許多使用者會在週末用筆電打開做月度規劃。

### 7.5 深淺模式

建議支援 light / dark 雙模式。深色模式對於「在床上睡前看一眼」的場景很重要,但兩種模式都應該保持產品的「平靜、克制」調性。

## 8. 特殊設計挑戰

### 8.1 Dashboard 的資訊密度

Dashboard 要在小小的螢幕上顯示 6 個 Block 的關鍵資訊。挑戰:

- 不能讓使用者覺得「資訊爆炸」
- 不能讓任何單一 Block 占據太多空間
- 必須能在 1 屏內看到最重要的 2–3 個 Block(儲蓄率、淨資產、健康警示)
- Mobile 上各 Block 之間的視覺切割要清楚

### 8.2 信用卡帳單匯入的審核 UX

Step 4 是整個產品最複雜的互動頁面。挑戰:

- 一次可能有 30–80 筆交易
- 「已自動分類」與「待審核」要視覺區隔
- 改分類操作要輕量(理想:點 tag → 跳出選項 → 一指選擇)
- 批次操作要直覺
- 完成感要明確

### 8.3 預算進度條的「中性」表達

預算用量達 95% 時,如何不用紅字、不用驚嘆號,還能讓使用者感到「該注意了」?可探索方向:

- 色階深淺(從淡到深,但都在同一色相)
- 留白變化
- 字重變化
- 用「剩餘可用日均」的數字傳達緊迫感而不靠色彩

### 8.4 配置餅圖的「現況 vs 目標」對比

兩個比例要並排對比,挑戰:

- 餅圖並排視覺上不夠對稱
- 偏離度的視覺化(哪一塊多、哪一塊少)需要直覺
- Mobile 上的縮放挑戰

可考慮的方案:對稱長條圖、Sankey 圖、或自定義的混合視覺化。

### 8.5 健康警示卡片的優先級表達

健康警示有 4 個等級(嚴重 / 警示 / 教育 / 資訊),但**全都不能用紅黃綠**。挑戰:

- 用什麼視覺語言區分嚴重度?
- 如何讓使用者一眼分辨優先級?
- 卡片標題的字重、icon 形狀、留白、位置都是工具

### 8.6 月度報告的調性

這是使用者「對自己這個月下評語」的時刻。挑戰:

- 不論結果好壞,如何維持中性?
- AI 解讀文字塊與數據塊的視覺整合
- 免責聲明的位置與設計(必要但不喧賓奪主)
- 里程碑(若有達成)的視覺處理(慶祝但不誇張)

### 8.7 隱私模式的視覺切換

切換為隱私模式時,所有金額變 ¥¥¥,但結構、比例、走勢仍可見。挑戰:

- 切換動效要平順
- 使用者必須清楚自己在隱藏模式
- 走勢圖的縱軸如何處理(數字也要遮?)

### 8.8 等寬數字的對齊

所有金額都要用等寬數字,讓位數對齊。挑戰:

- 字型選擇(很多漂亮字型沒有 tabular figures)
- 中文 + 數字的混排對齊
- 不同字級下的視覺一致性

## 9. 技術與合規約束

### 9.1 i18n 基礎

Phase 0 僅繁中,但設計上應預留 i18n:

- 文字長度可變(英文通常更長,日文相近)
- 不在 icon 內嵌入文字
- 日期、貨幣、數字格式要可配置

### 9.2 可訪問性 (a11y)

- WCAG AA 對比度
- 鍵盤完整可操作
- Screen reader 友好
- **不只靠顏色傳達狀態**(色盲使用者要能看懂優先級,因此前述用「形狀 + 色階」而非紅黃綠就更顯重要)

### 9.3 隱私模式的全域影響

設計每一個顯示金額的元件時,都要考慮「啟用隱私模式時的呈現」。這不是可選功能,是基礎能力。

### 9.4 法規必要元素

以下元素**必須**出現在指定位置:

| 元素 | 出現位置 |
|---|---|
| 「本服務不提供投資建議」 | 註冊頁勾選框 |
| 免責聲明 | 每份 AI 解讀文末 |
| 「本資訊僅供參考,不構成投資建議」 | 月度報告 footer |
| 隱私政策、使用者條款連結 | Footer 永久顯示 |

設計上要找到「合規必要 + 不喧賓奪主」的平衡。

### 9.5 技術 stack(影響元件實作)

- Next.js 14+ (App Router)
- Tailwind CSS
- shadcn/ui(可作為基礎元件起點,但需深度客製以符合本產品調性)
- Recharts 或自訂 SVG(圖表)

## 11. 檢驗清單

設計交付前,請對照以下清單自我檢驗。任何一條不通過都應該重新檢視設計決策。

### 11.1 哲學檢驗

- [ ] 主畫面第一視覺焦點是儲蓄率,不是總資產?
- [ ] 是否完全沒有紅字超支警告?
- [ ] 是否完全沒有連勝紀錄、streak、排行榜?
- [ ] 預算進度條給使用者的感受是「掌控」而非「監視」?
- [ ] 月度報告即使在儲蓄率低的情況下,也不會讓使用者感到失敗?

### 11.2 功能檢驗

- [ ] 信用卡帳單匯入流程能在 5 分鐘內完成 50 筆交易的審核?
- [ ] Onboarding 能在 5 分鐘內完成且每步可跳過?
- [ ] 隱私模式切換無需確認彈窗、即時生效?
- [ ] 大額手動記錄能在 30 秒內完成?

### 11.3 系統檢驗

- [ ] 所有金額用等寬數字、位數對齊?
- [ ] 健康警示優先級能在不依賴紅黃綠的情況下被分辨?
- [ ] 設計支援 light / dark 雙模式?
- [ ] WCAG AA 對比度達標?
- [ ] 響應式斷點下三種設備都可用?

### 11.4 合規檢驗

- [ ] 註冊頁有「不提供投資建議」勾選框?
- [ ] AI 解讀末尾必有免責聲明?
- [ ] 月度報告 footer 必有免責語?

## 12. 開放問題(留給設計端發揮)

以下問題刻意不在本文件中決定,設計端可自由探索:

1. **產品名稱與品牌調性**:NetWorth Cockpit 只是內部代號,正式品牌名與調性可重新發想。台灣市場、信任感、成熟感是關鍵字。
2. **主色選擇**:需要傳達「金錢但不庸俗」、「成熟但不老氣」、「平靜但不無聊」。是綠色?藍色?中性灰?或意想不到的選擇?
3. **健康警示的視覺語言**:四階優先級的色階具體該長什麼樣?
4. **資產類別識別色**:股/債/現金/加密 各用什麼色?要與預算類別色協調。
5. **配置對比的視覺化**:餅圖 vs 長條 vs 自創?
6. **里程碑的慶祝方式**:輕但有感、不誇張、不打斷使用者的方式是什麼?
7. **空狀態的擬人化程度**:完全文字、icon、插畫?要避免幼稚化但也不能太冷。
8. **動效的節制範圍**:哪些互動值得動效、哪些不該有?
